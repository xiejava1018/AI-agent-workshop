# 产品需求规格说明书（PRD）—— AI Agent Workshop

> 版本：v0.1（草案，待评审）
> 日期：2026-07-16
> 范围依据：与用户校准确认（务实版多用户工作台 / 内部自用 / 会话·配额级隔离 / Vue3 主界面+管理台 / 本期重点 多 Agent 编排 + MCP 工具生态）

---

## 1. 产品概述

### 1.1 定位
**AI Agent Workshop** 是一个**内部自用的多用户 AI Agent 工作台**。用户在注册/被管理员开通后，登录进入自己独立的工作空间，调用基于 Pi Agent 的 Agent 能力完成编码、分析、自动化等任务；管理员通过统一后台管理用户、团队与配额。

### 1.2 一句话定义
> 一个团队共享、可治理的 AI Agent 工作台：成员在团队工作空间内既能和单个 Agent 对话，也能编排多个 Agent 协作；工具能力通过 MCP 无限扩展。

### 1.3 与 MateClaw / OpenClaw 的关系
- 借鉴 MateClaw「数字员工 / 数据不出门 / 可治理」的产品理念，以及 OpenClaw 已验证的 Pi Agent 产品层模式。
- **本期为务实版**：不追求 MateClaw 全套（多模态、8 渠道 IM、工作流 DSL、触发器、知识库、企业级 RBAC、信创），但架构为后续扩展留好接口（Supervisor、MCP、Skill 技能机制、Extension 机制）。

## 2. 目标与价值

| 目标 | 说明 |
|---|---|
| 能力收敛 | 把分散的 AI 编码/对话能力收口成一个团队可共享、可治理的工作台 |
| 隔离空间 | 每个团队拥有隔离的工作空间（团队内共享项目目录）；会话与配额按用户隔离，互不干扰 |
| 可治理 | 管理员可开通账号、配置配额、查看审计，避免资源滥用 |
| 可扩展 | 通过多 Agent 编排与 MCP，能力可随业务增长平滑扩展 |

## 3. 用户角色与场景

### 3.1 角色
| 角色 | 职责 | 入口 |
|---|---|---|
| 普通用户 | 在自己的工作空间调用 Agent 干活（单 Agent 对话、发起多 Agent 任务、使用 MCP 工具） | Vue3 用户主界面 |
| 管理员（OWNER/ADMIN） | 开通/停用用户、管理团队、配置配额、查看审计与监控 | Vue3 管理后台 |

### 3.2 典型场景
1. **个人编码助手**：用户登录后，在工作台与编码 Agent 对话，生成/修改代码，会话与文件隔离在本空间。
2. **团队共享 Agent**：管理员/OWNER 把团队常用技能与 MCP 预装到团队作用域，团队成员创建的数字员工默认继承；特定数字员工（如"编码 Agent"）可额外绑定专属工具，开箱即用。
3. **多 Agent 协作**：用户发起一个复杂任务，系统用 Supervisor 拆给多个子 Agent（如「调研 Agent」「编码 Agent」「审查 Agent」）——每个子 Agent 都是一个数字员工（Agent）实体，自动获得其绑定的技能/MCP，并行/串行协作汇总结果。
4. **管理员治理**：开通新同事账号、设置每人每日 token 上限与并发数、排查异常审计。
5. **技能复用**：用户在处理「生成周报」「导出 PPT」「代码审查」等高频任务时，调用对应 Skill；管理员把团队常用 Skill 预装到团队作用域，成员开箱即用。

### 3.3 角色与权限矩阵
两级角色易混淆，明确如下（详见设计文档 §3.3）：

| 维度 | 平台管理员 | 团队 OWNER | 团队 ADMIN | 团队 MEMBER |
|---|---|---|---|---|
| 用户开通/停用 | ✅ | ❌ | ❌ | ❌ |
| 创建团队 | ✅（也可） | ✅（自建即 OWNER） | ❌ | ❌ |
| 团队成员/角色管理 | 全部团队 | 仅本团队 | 仅本团队 | ❌ |
| 团队配额设置 | ✅ | ✅ | ❌ | ❌ |
| 安装全局精选技能 | ✅（仅无凭证） | ❌ | ❌ | ❌ |
| 配置带凭证 MCP | 平台级全局公共服务；团队级由 OWNER/ADMIN 在本团队配 | ✅本团队 | ✅本团队 | ❌ |
| 读其他团队会话/凭证 | ❌（仅治理视图，仅可看会话元数据：标题/时间/token 用量，不可读会话正文） | ❌ | ❌ | ❌ |
| 创建/使用数字员工 | 可 | 本团队+个人 | 本团队 | 个人+团队预置 |

> 铁律：任何角色均不可读其他团队的会话内容与 MCP 凭证；团队 ADMIN ≠ 平台管理员。

## 4. 功能需求（分期）

### 4.1 MVP（本期交付）

#### 4.1.1 用户与认证
- FR-01 注册：本期不开放公开注册；仅管理员（root/admin）可创建用户，初始密码强制首次登录修改。
- FR-02 登录：用户名 + 密码登录，返回 access token（短期）+ refresh token（长期，cookie 存储）。
- FR-03 续期：access token 过期用 refresh token 无感轮换（jti 黑名单防重放）。
- FR-04 改密：新用户首次登录强制改密；普通用户可自助改密。

#### 4.1.2 工作空间与隔离
- FR-05 空间模型：采用 **Team / Project** 两级（沿用现有模型）；**会话、配额按用户/团队隔离（会话·配额级隔离）**；文件为**团队共享项目目录**（运行时 `cwd = user.lastProjectId → Project.rootPath`），非每用户独立文件系统。**Session 同时挂 `userId`（会话隔离）+ `projectId`（cwd 指向团队共享目录）；同团队多用户共用同一目录、各自独立会话，本期不对文件并发写做应用层互斥（内部自用、低并发），仅审计层记录写操作，M5 视情况引入文件锁（B2 取舍，2026-07-16）**。
- FR-06 配额：每用户默认最多 5 个**同时活跃会话**、全局 50 兜底；可按团队配置 token 日上限与「并发数」（**本期并发数即同时活跃会话数，不另设独立并发维度**）；多 Agent 编排的委派子会话走专用池、仅根会话计入该上限（见 FR-12）。
- FR-07 防泄漏：所有数据访问强制按租户上下文过滤，租户 ID 由服务端会话推导，不信任客户端传入。

#### 4.1.3 Agent 对话工作台（Vue3 主界面）
- FR-08 对话：用户在工作台与单个 Agent 对话，支持流式输出（SSE/WS）。
- FR-09 会话管理：新建/切换/删除会话；会话持久化（JSONL，复用 Pi Agent SessionManager）。
- FR-10 工具可见：对话中可触发内置工具（读/写/编辑/检索/bash 等）与已绑定的 MCP 工具。

#### 4.1.4 多 Agent 编排（本期重点）
- FR-11 Supervisor：用户可发起「多 Agent 任务」，由 Supervisor 将任务拆解为子任务，委派给动态创建的子 Agent。
- FR-12 委派模式：支持同步 / 并行 / 异步三种委派（并行上限 8，对标 MateClaw）；子 Agent 委派树深度限制（默认 3 层）。**委派子会话走专用池、仅根 Supervisor 会话计入用户会话上限**，避免 8 并行 + 1 父撞每用户 5 / 全局 50。
- FR-13 结果回收：子 Agent 的结果回流 Supervisor，汇总后返回用户，过程可在界面可见（树状/时间线）。
- FR-14 持久化：委派树结构由 `DelegationTree` 表持久化（记录 root/parent/child 会话与 mode/depth），支持界面树状展示与断点恢复；SessionManager.branch/fork 仅单会话历史分支，不表达跨会话父子树。

#### 4.1.5 MCP 工具生态（本期重点）
- FR-15 接入：引入社区包 `pi-mcp-extension`，支持 stdio / SSE / Streamable HTTP 三种传输接入 MCP Server。
- FR-16 绑定：MCP Server 可按 **团队 / Agent** 维度绑定；不同团队的 Agent 看不到彼此的 MCP 工具（不跨租户泄漏）。
- FR-17 管理：管理员在后台配置可用 MCP Server 列表与绑定关系。

#### 4.1.6 技能（Skill）调用能力（本期重点）
Agent 可调用 **Skill（技能）**——以 `SKILL.md`（YAML 元数据 + 指令）形式封装的可复用能力包，对标 OpenClaw / Claude 技能体系。**本仓库后端已具备技能安装 / 搜索 / 解析雏形**（`/api/skills/install`、`/api/skills/search`、`/api/plugins`，以及 Pi Agent 的 `Skill` 类型与 `parseSkillBlock`），本期需补齐**多租户绑定**与**前端界面**，使其成为一等公民能力。
- FR-18 技能库：展示当前用户 / 团队可见的技能列表（名称、描述、来源、作用域、启用状态）。
- FR-19 市场与安装：在界面内搜索 skills.sh 市场（`/api/skills/search`），一键安装到「团队」或「全局」作用域（`/api/skills/install` / `/api/plugins`）。
- FR-20 调用方式：① 显式——对话中输入 `/<skill>` 斜杠命令（对应 `disableModelInvocation: true` 的技能）；② 模型自决——Agent 在推理中识别需求后自动触发技能（`<skill>` 块经 `parseSkillBlock` 解析并注入指令）。
- FR-21 多租户绑定（隔离模型已决策）：默认 **团队隔离 + 全局精选**——`global` 仅管理员安装无凭证通用技能、全员可用；`team` 为默认隔离边界（成员共享、团队间不可见）；`user` 为可选个人层。某会话仅能调用本 tenant 已启用技能，不跨租户泄漏；作用域由服务端按 tenant 上下文推导，禁止客户端指定。**个人安装技能不强制属于某团队（`userId` 即可），纯指令型技能可完全 user 独立安装；但若含凭证 / 绑定 MCP，则仍受 MCP 凭证隔离铁律约束（B1 边界，2026-07-16）**。

#### 4.1.7 管理后台（Vue3）
- FR-22 用户管理：列表、创建、停用、改密、配额设置。
- FR-23 团队管理（混合生命周期，已决策 2026-07-16）：
  - 用户可**自建团队**，自动成为该团队 OWNER；
  - 管理员（OWNER/ADMIN）可把任意用户**拉入**团队并赋角色（MEMBER/ADMIN）；
  - 团队可生成**邀请链接**（带时效/一次性 token，可选审批），被邀请者点击即加入（**邀请链接定位为"把已有平台账号的跨团队/跨组织成员拉入本团队"的自助入口，不面向全新外部人**；全新外部人须走"管理员开通账号 → OWNER 发邀请 → 用户接受加入"路径。链接不创建账号，被邀请者须先有平台账号，B4 澄清 2026-07-16）；
  - 成员可退出，OWNER 可移交所有权；
  - 功能含：团队列表、成员管理（角色/移除）、团队级配额（token 日上限/并发）、邀请链接管理。
- FR-24 审计与监控：操作审计日志、会话 / 配额使用大盘（ECharts）。
- FR-25 数字员工（Agent）管理（绑定单元，已决策 2026-07-16）：
  - **两类数字员工**：① 团队数字员工（`teamId` 必填，成员经所属 Team 间接使用）；② 个人数字员工（`ownerUserId` 必填、仅自己可用、能力受团队授权约束）。
  - 创建/配置数字员工：名称、描述、systemPrompt、默认模型、归属（团队/个人）；
  - 绑定技能与 MCP：为该数字员工**显式授予白名单或排除**，实现"编码 Agent 用代码技能、写作 Agent 用 PPT 技能"的精细分工；
  - 有效作用域四层继承（global 精选 → team 预装 → user 覆盖 → agent 收敛，详见设计文档 §5.4）；团队/个人预装的技能/MCP 默认被继承，可逐 Agent 覆盖；
  - 多 Agent 编排中的子 Agent 即一个 Agent 实体实例（Supervisor 经「可用数字员工目录」发现它），自动获得其绑定。
  - **后端路由须避开现有 `/api/agent/*`（会话端点）**，数字员工用 `/api/digital-employees`（见设计文档 §5.4 / §10）。
- FR-26 模型与凭证配置（双层，已核实代码 2026-07-16）：
  - **管理员配置**：可用模型清单、默认模型、故障回退顺序，以及平台级 API Key 池（共享额度），在管理后台「模型配置」界面维护；
  - **用户 BYOK**：个人可在「我的设置」为各厂商填 API Key，调用时优先于平台密钥；未填则走平台共享额度；
  - 故障自动回退：主模型不可用时按管理员配置顺序切换，用户可开关。

## 5. 非功能需求

| 类别 | 要求 |
|---|---|
| 隔离 | 会话·配额级隔离（应用层强制 tenant 过滤）；SQLite 当前可接受，Postgres+RLS 为后续可选项 |

| 可维护 | TS 单仓（pnpm monorepo）；双前端清晰分层；OpenSpec 驱动变更 |
| 合规 | 内部自用，数据不出门；许可证守住已用 MIT（pi-web / Pi Agent） |

## 8. 验收标准（Definition of Done）

1. **多用户独立**：多个用户可同时登录，会话、配额、文件互不干扰，无法越权读取他人数据。
2. **Vue3 主界面**：用户可在 Vue3 界面发起 Agent 对话，输出以流式展示，会话可持久化与切换。
3. **多 Agent 编排**：用户可发起多 Agent 任务，Supervisor 正确拆解/委派/汇总，过程在界面可见，委派树可持久化。
4. **MCP 工具**：管理员可配置 MCP Server 并绑定到团队/Agent；子 Agent 能调用绑定工具，且工具不跨团队泄漏。
5. **管理后台**：管理员可管理用户/团队/配额，查看审计与监控大盘。
6. **安全基线**：双 token 续期正常；per-user session cap 生效；危险操作有审计留痕。
7. **技能能力**：用户可在界面浏览 / 安装 / 启用技能；对话中可通过 `/<skill>` 或模型自决触发技能；技能按团队隔离，不跨租户泄漏。
8. **模型与凭证**：管理员可在「模型配置」维护可用模型清单 / 默认模型 / 回退策略与平台密钥池；用户可在「我的设置」填个人 API Key（BYOK）优先于平台密钥；主模型故障自动回退生效。

## 9. 假设与依赖

- 假设 `pi-mcp-extension` 在 Node 22 + Pi Agent 0.80.6 下可安装并正常工作（待 Spike 验证）。
- 假设 `apps/web` 现有 `/api`（含 agent 流式接口）可被 Vue3 前端消费（待验证代理/跨域）。
- 依赖 Pi Agent 引擎持续可用（MIT，已验证 0.80.6 API 面）；其 `Skill` 类型、`parseSkillBlock` 与 `loadSkills` 已核实，技能机制可落地。
- 本仓库 `apps/web` 已有技能后端雏形（`/api/skills/install`、`/api/skills/search`、`/api/plugins`），需补多租户作用域与前端界面。
- 模型配置：现有 `apps/web/app/api/models-config` 已把配置写成 `agentDir/models.json`（文件式）；本期「模型配置」UI 复用该文件式配置为单真相源，避免与 `ModelConfig` 表双真相源冲突（见设计文档 §10）。
- 数字员工实体（Agent）后端路由须避开现有 `/api/agent/*`（会话端点），避免会话/数字员工概念混淆（见设计文档 §5.4）。

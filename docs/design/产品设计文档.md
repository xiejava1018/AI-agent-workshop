## 1. 架构总览

### 1.1 统一前端 + 后端运行时架构（已与用户确认）
```
┌─────────────────────────────────────────────────────────────┐
│  apps/dashboard (Vue3 + Element Plus + Pinia + Vue Router)    │
│  角色：用户主界面（Agent 对话 / 多 Agent 编排 / 工作空间）        │
│        + 管理后台（用户/团队/配额/审计/监控）                    │
│  通过 vite proxy '/api' → apps/web 后端                         │
└───────────────────────────┬─────────────────────────────────┘
                              │  /api  (REST + SSE/WS 流式)
┌───────────────────────────▼─────────────────────────────────┐
│  apps/web (Next.js 16 / React 19)  =  后端 API + Agent 运行时   │
│  · Next.js Route Handlers 暴露 REST /api/* 与流式端点           │
│  · lib/rpc-manager.ts：Pi Agent 会话工厂（createAgentSession*） │
│  · lib/team-auth.ts：多租户上下文推导与隔离                      │
│  · lib/tool-presets.ts：工具预设（内置 + MCP）                   │
│  · Prisma + SQLite（后续可迁 Postgres+RLS）                     │
│  · Pi Agent 0.80.6：AgentSession / SessionManager / ModelRegistry│
│    / AuthStorage(InMemory) / defineTool / Extension            │
└─────────────────────────────────────────────────────────────┘
```

**关键修正**：Vue3 不再是"纯管理台"，而是**用户面对 Agent 的主界面 + 管理后台**；React 那层（apps/web）定位为后端 API 与 Agent 运行时宿主，其自带 React UI 可保留为开发/参考界面。

> **重要澄清（2026-07-16）**：本方案的"双前端"指 monorepo 里的**两个应用**——`apps/dashboard`（Vue3，唯一面向用户的 UI）与 `apps/web`（Next.js，后端 + Agent 运行时），**绝非"两个面向用户的界面"**。系统实现是**单一统一的前端应用 + 单一入口**，所有用户（普通用户 / 管理员 / 团队 OWNER）登录后进入同一个界面；菜单按 RBAC 权限矩阵（§3.3）渲染——管理员额外看到"平台管理"菜单组，团队 OWNER 额外看到"团队"菜单组，普通用户只看到"工作区 / 我的资源"。后端对每个 API 亦按权限校验（纵深防御），不仅靠前端点隐藏菜单。
> 因此 `docs/ui-design/user-view.html` 与 `admin-view.html` **只是同一套统一前端在两种角色下的"设计视角"**，为评审方便而拆分绘制，**不是要分别开发的两个工程**；真正的整体蓝图见 `docs/ui-design/index.html`（单一入口 + 角色切换演示）。

## 3. 多租户与隔离模型

- **模型**：沿用现有 `Team / TeamMember / Project / User / Role` 模型（会话·配额级隔离）。新增一等公民 **`Agent`（数字员工）**：Agent 归属 Team，是 **skills / MCP 的绑定单元**（见 §5.4）；Team 拥有多个 Agent，用户成员通过所属 Team 间接使用 Agent。
- **上下文推导**：所有请求经 `middleware.ts` + `lib/team-auth.ts` 解析出 `tenantId / userId / teamId`，后续 DB 查询与 Agent 调用强制带该上下文。
- **会话与项目目录绑定（B2 取舍，2026-07-16）**：`Session` 同时挂 `userId`（会话隔离）+ `projectId`（`cwd` 指向团队共享项目目录 `Project.rootPath`）。同团队多个用户共用同一 Project 目录、各自起独立会话写同一目录——**本期不对文件并发写做应用层互斥**（内部自用、低并发场景），仅在审计层记录写操作；后续 M5 视情况引入文件锁 / 分支隔离。该取舍须显式确认，避免实现时各人理解不一。
- **配额**：`lib/session-cap.ts` 已实现 `Map<userId, count>`（每用户默认 5，全局 50 兜底）；后续扩展 token 日上限、并发数。
- **防泄漏铁律**：tenantId 一律由服务端推导，绝不信任客户端传入；所有查询强制按 tenant 过滤。

### 3.1 Team 生命周期（混合模式，已决策 2026-07-16）
- **用户自建团队**：登录用户可创建 Team，自动成为 `ownerUserId`（OWNER 角色），无需管理员介入。
- **管理员拉人**：管理员（OWNER/ADMIN）通过 `TeamMember` 将任意用户加入团队，并赋予角色（MEMBER/ADMIN）。
- **邀请链接（B4 澄清，2026-07-16）**：团队可生成邀请链接（带一次性/时效 token，可选审批），被邀请者点击即加入。**定位为"把已有平台账号的跨团队 / 跨组织成员拉入本团队"的自助入口（免去管理员逐个拉人），不面向全新外部人**——被邀请者须先已有平台账号或由管理员先开通，链接本身不创建账号，否则入职流程断在账号开通这一步。全新外部人入职路径：管理员开通账号 → OWNER 发邀请 → 用户接受加入。
- **退出 / 移交**：成员可主动退出；OWNER 可把所有权移交给其他成员。
- 这与 MateClaw「员工自建数字员工」的自助理念一致，同时保留管理员管控能力。

### 3.2 模型与凭证配置（双层，已决策 2026-07-16）
> 经代码核实（`app/api/models/route.ts`、`ModelRegistry`、`app/api/auth/api-key/[provider]`）：模型能力是**双层**配置，并非"单纯管理员配"。

- **管理员层（全局资源）**：通过 `ModelRegistry` + `modelsPath`（models 配置文件）配置「可用模型清单 + 默认模型 + 故障回退顺序」；并可在后台维护**平台级 API Key 池**，供未带个人密钥的用户走共享额度。当前落地为后端配置文件，本期补**管理后台「模型配置」界面**（见 UI 设计稿 `admin-view.html`）。
- **用户层（BYOK）**：每个用户可在自己的设置里为各 provider 填 API Key（`authStorage.set(provider, { key })`）；`ModelRegistry` 按用户 `authStorage` 实例化，故**带个人密钥的厂商优先用个人 key，未填才走平台共享额度**。
- **为何契合「团队隔离 + 全局精选」**：模型清单当作全局精选资源由管理员统一管控；凭证则按人/按团队隔离（BYOK），不跨租户共享——与 §7 的 MCP 凭证隔离原则同源。
- **故障自动回退**：主模型不可用时按管理员配置的回退列表自动切换；用户侧可开关。

### 3.3 角色与权限矩阵（新增，安全边界）
两级角色易混淆，明确如下：

| 维度 | 平台管理员（admin/root） | 团队 OWNER | 团队 ADMIN | 团队 MEMBER |
|---|---|---|---|---|
| 用户开通/停用 | ✅ | ❌ | ❌ | ❌ |
| 创建团队 | ✅（也可） | ✅（自建即 OWNER） | ❌ | ❌ |
| 团队成员管理/角色 | 全部团队 | 仅本团队 | 仅本团队（无所有权移交） | ❌ |
| 团队配额设置 | ✅ | ✅ | ❌ | ❌ |
| 安装全局精选技能 | ✅（仅无凭证） | ❌ | ❌ | ❌ |
| 配置带凭证 MCP | 平台级全局公共服务 ✅；**团队级 MCP 由 OWNER/ADMIN 在本团队配，绝不跨团队** | ✅（本团队） | ✅（本团队） | ❌ |
| 看到其他团队数据 | ❌（仅治理视图，**仅可看会话元数据**：标题/时间/token 用量，用于治理审计；**不可读会话正文内容**） | ❌ | ❌ | ❌ |
| 创建/使用数字员工 | 可 | 本团队 + 个人 | 本团队 | 个人 + 团队预置 |

> 铁律：任何角色均**不可读其他团队的会话内容与 MCP 凭证**；团队 ADMIN ≠ 平台管理员，不能跨团队操作。**平台管理员亦仅可看会话元数据（标题/时间/token 用量）用于治理审计，不可读任意用户的会话正文内容**——会话正文仅会话所属用户本人可见（D2 隐私取舍，2026-07-16）。

## 4. Agent 运行时层（接缝设计）

`apps/web/lib/rpc-manager.ts` 已是 Pi Agent 集成核心，已使用：
- `createAgentSessionServices({ cwd, agentDir })` — 构建 AuthStorage / ModelRegistry / SettingsManager 等服务。
- `createAgentSessionFromServices({...})` — 用注入的服务创建会话。
- `SessionManager` — JSONL 持久化、branch/fork。

**本层设计要点**：
- 会话工厂对外暴露统一接口 `spawnSession(opts)`，opts 内含 `tenantId / userId / teamId / agentId / systemPrompt / model`。**skills 与 MCP 工具不再在此硬编码**，而是由 `agentId`（所属的数字员工）经四层解析（global → team → user → agent）得出有效作用域后，通过 `resourceLoaderOptions` 注入（见 §5.4、§6.3、§7.3）。
- 注入机制（**已对照 SDK 0.80.6 实际签名修正**）：技能经 `createAgentSessionServices` 的 `resourceLoaderOptions.skillsOverride` / `additionalSkillPaths` / `noSkills` 注入；MCP 扩展经 `resourceLoaderOptions.extensionsOverride` / `additionalExtensionPaths` 注入；会话层 `createAgentSessionFromServices` 仅接受 `tools: string[]` / `customTools` / `excludeTools` 作工具级约束。**注意：SDK 的 `createAgentSessionServices` 并没有 `skills`/`tools` 平铺参数**，原"注入 skills 列表"的写法是错的，必须走 resourceLoaderOptions。SDK 依然零改动。
- **⚠️ 缓存键修正（避免 per-agent 绑定被击穿）**：`lib/rpc-manager.ts` 现有 `getOrCreateServices(cwd, agentDir)` 按 `cwd` 缓存并复用同一 `resourceLoader`（且 `agentDir = getAgentDir()` 为全局），同一 Project（cwd）下不同数字员工会拿到同一份技能/扩展，**导致按 agentId 的绑定形同虚设**。须把缓存键改为纳入「技能/扩展作用域 hash」或按作用域重建 services（例如 `getOrCreateServices(cwd, scopeHash)`），否则绑定不生效。
- 每次 spawn 注入**按租户的** `SettingsManager`（cwd/agentDir 按团队或用户区分）与 `AuthStorage`（InMemory 后端，按请求注入 BYOK 密钥）。
- 流式：会话事件经 `session.subscribe` 收集，由 Next.js 流式端点（SSE/WS）推给 Vue3。

## 5. 多 Agent 编排设计（Supervisor）

### 5.1 现状
Pi Agent **无原生子 Agent / Supervisor**（已核实）。必须自建。

### 5.2 组件
```
User
  │ 发起「多 Agent 任务」
  ▼
Supervisor (协调者，本身是一个 AgentSession)
  │ 用 ReAct 或 Plan-and-Execute 拆解任务
  ▼
DelegateAgentTool (一个 defineTool)
  │ 内部调用 createAgentSessionFromServices 创建子 Agent
  ├─ 子 Agent A（如：调研）
  ├─ 子 Agent B（如：编码）   ← 支持 同步 / 并行 / 异步
  └─ 子 Agent C（如：审查）
  │ 子 Agent 结果经 session.subscribe 回流
  ▼
Supervisor 汇总 → 返回用户
```

### 5.3 关键设计决策
- **委派协议**：实现一个 `DelegateAgentTool`（标准 `defineTool`），其执行体在 Node 端新建一个子 `createAgentSessionFromServices` 实例并 `prompt()`，等待结果后返回。天然复用 Pi Agent 工具机制。
- **Supervisor 如何发现子 Agent（新增）**：`DelegateAgentTool` 在注入时由 `spawnSession` 按当前 `teamId`（个人数字员工按 `ownerUserId`）查出「可用数字员工目录」（Agent 实体列表）作为候选上下文传给 Supervisor；LLM 据此涌现式决定委派给哪个数字员工。**不可仅靠用户裸打字枚举 Agent**，否则编排不可控且可能越权。子 Agent 沿用目标 Agent 实体的 systemPrompt / skills / MCP 绑定（见 §5.4）。
- **深度限制**：委派树默认最深 3 层（对标 MateClaw）；Supervisor 在拆解时校验深度，超出则改为单 Agent 直答或报错。
- **委派模式**：本期实现**同步 / 并行 / 异步**三种委派（对应 MateClaw 的 `delegateToAgent` / `delegateParallel` / `delegateAsync`，实测参数见 §5.5）；异步以 `task_id` + `taskOutput` 回填，UI「多 Agent 编排」屏支持模式切换。并行上限 8（对标 MateClaw `delegateParallel`）。
- **状态与错误**：每个子 Agent 独立 try/catch；失败不影响兄弟节点（可配置 fail-fast / best-effort）；Supervisor 负责错误聚合与重试。
- **会话配额口径（修正，避免撞上限）**：并行委派 8 子 Agent + 1 Supervisor = 9 个会话，会直接撞每用户默认 5 / 全局 50 上限，且 LRU 淘汰可能误杀父 Supervisor。规则：**仅根 Supervisor 会话计入用户的 per-user 会话上限**；派生子会话走「委派专用池」（独立计数、不计入 5/50、带 depth/owner 归属），其 token 用量按 §5.5 滚回根会话统一计费。
- **持久化（修正）**：`SessionManager.branch()/fork()` 只是**单会话内部的历史分支**，并非跨会话父子树，不能用来表示 Supervisor→子 Agent 的委派关系。须自建 `DelegationTree`（已在 §10 列出）记录 `rootSessionId / parentSessionId / childSessionIds / mode / depth / status`，支持界面树状展示与断点恢复。
- **可见性**：子 Agent 的执行事件（tool_update / message）经 Supervisor 转发，Vue3 以树状/时间线展示。

### 5.4 Agent（数字员工）实体：绑定的基本单元
- **实体**：新增 `Agent`（数字员工）一等公民——`id / teamId(nullable) / ownerUserId / name / description / systemPrompt / model / scope(team|personal) / createdAt`。**单一 `Agent` 表承载所有数字员工，不新增独立的 `AgentTemplate` 实体（YAGNI，2026-07-16）**；"团队模板"= `scope=team` 的 Agent（团队只读继承，成员不可改），"克隆为个人"= 复制一行新 `scope=personal` Agent 并允许改写绑定。
  - **团队数字员工**：`teamId` 必填、`scope=team`，归属某团队，成员通过所属 Team 间接使用；可"设为团队默认"（即团队预装层快捷操作，非新实体）。
  - **个人数字员工**：`teamId` 可空、`ownerUserId` 必填、`scope=personal`，仅创建者自己可用；其能力以"全局精选 + 本团队已授权"作底座，个人安装的**纯指令型技能可独立于团队**（userId 即可），但若绑定带凭证的 MCP 则仍受 §7.3 团队凭证隔离铁律约束（越权绑定被服务端拒绝）。
- **绑定单元**：skills 与 MCP **绑定到 Agent**（沿用 MateClaw 的 `AgentSkillBinding` / `AgentToolBinding` 思路），而非直接绑 Team。Team 层只做"预装/精选"，Agent 层决定该数字员工实际能用哪些。
- **有效作用域解析（四层继承）**（解析某 Agent 的有效技能/MCP）：
  1. `global`（全站精选库）——仅管理员安装的无凭证通用能力，全员底座；
  2. `team`（团队预装层）——团队默认继承；
  3. `user`（个人覆盖层，仅 personal 数字员工 / 个人会话生效）——个人安装的技能/MCP；**个人安装技能不强制属于某团队（`userId` 即可），纯指令型技能可完全 user 独立安装；但若含凭证 / 绑定 MCP，则仍受 §7.3 团队凭证隔离铁律约束**（B1 边界，2026-07-16）。
  4. `agent`（数字员工最终绑定单元）——在此收敛。
  - 解析顺序：global → team → user → agent；`agent` 层的 mode 决定最终生效集（inherit=继承以上 / include=仅白名单 / exclude=显式禁用）。**绝不跨 tenant 推导，禁止客户端指定目录。**
- **与多 Agent 编排天然统一**：编排中的「子 Agent」即一个 `Agent` 实体实例（Supervisor 经 §5.3 的「可用数字员工目录」发现它），自动获得其绑定的技能/MCP，无需额外配置。
- **命名空间**：新增数字员工实体后端路由须避开现有的 `/api/agent/[id]`、`/api/agent/new`（它们是**会话**端点），建议用 `/api/digital-employees`（或 `/api/agents-catalog`），避免会话/数字员工混淆（见 §10）。
- **管理 UI**：见 UI 原型「数字员工」屏（创建/配置数字员工、绑定技能与 MCP）。

### 5.5 参考 MateClaw 实测（编排三机制与安全护栏，2026-07-16）
> 已读 `mateclaw-server` 真实源码（`tool/builtin/DelegateAgentTool.java` ≈1200 行、`planning/service/PlanningService.java`、`resources/docs/workflow.md`、`tools.md`）。MateClaw 实际上提供**三套互补**的编排机制，均经其生产代码验证；我们的自建方案可直接采用其踩坑定下的参数与护栏，避免重复踩坑。

**A. 委派（Delegation，最常用）**——暴露三个工具，由 LLM（Supervisor）涌现式决定调用：
- `delegateToAgent(agentName, task, inheritParentContext?)`：串行委派，隔离会话，返回结果（默认继承父上下文）。
- `delegateParallel(tasks[])`：并行委派**最多 8 个**子 Agent（JDK 21 虚拟线程），有总超时（默认 300s 可配）、fail-fast（必需子任务失败即收束）、逐子完成事件回流。
- `delegateAsync(task)→taskId` + `taskOutput(taskId, block?)`：**异步解耦**，父 Agent 立即拿 `task_id` 继续推理，子 Agent 后台跑长任务（如 30 分钟调研），回头取结果；带归属鉴权（防跨用户猜 `taskId` 读他人结果）。

**B. 计划-执行（Plan-and-Execute，可选增强）**——`PlanningService` + 状态机：父 Agent 把目标拆成 `PlanEntity(goal, steps[])`，每步可指定 `assignedAgentId` 专项子 Agent，逐步委派执行；支持 `awaiting_approval` **审批门** + replay-resume（审批后续跑）。用于"有顺序、要人盯"的任务。

**C. 工作流（Workflow，v1.3.0+，后续扩展）**——极简线性：`steps` 数组 + 一个 `mode` 字段表达控制流（sequential/fan_out/collect/conditional/await_approval），**刻意不做 Dify 式 30 节点编排器**。本期不实现，但委派模型已为其留好接口（PRD §4.2）。

**安全护栏（MateClaw 踩坑定下，我们直接采用）：**
- 委派树深度上限 `MAX_DELEGATION_DEPTH = 3`；超出改单 Agent 直答或报错。
- 子 Agent 结果截断 `MAX_RESULT_LENGTH = 4000` 字符，防上下文爆炸。
- 子 Agent 禁用清单 `CHILD_DENIED_TOOLS`：`delegate*` / `listAvailableAgents`（防递归委派）、`remember*` / `forget_*`（防污染父记忆）、`setGoal*` / `completeGoal`（目标归父会话）、`create_employee`（防递归建团队提权）。
- 子 Agent **继承父 ChatOrigin**（channel/workspace/requester）；workspace 随子会话透传（= 我们的 Team 隔离），确保跨团队不可见。
- 事件回流：`registerBatchedRelay` + `broadcastObject(rootConv)`，子 Agent 的 tool_update/进度实时转发根会话，前端画成树/时间线（对应本文 §5.3 可见性）。
- 配额/计费滚回：`DelegatedUsageAccumulator` 把子 Agent token 用量**滚回根会话**（统一配额、计费），与本文 §3 配额模型一致。
- 身份转发：借鉴其 **on-behalf-of** 思路——把调用用户身份随委派/工具调用带上，供上游（含 MCP）做按人/按团队细粒度授权与审计（呼应 §7.3 on-behalf-of）。

**衔接**：§5.3 的三种委派模式即对应 A 的 serial/parallel/async；§5.4 的"子 Agent 即一个 `Agent` 实体实例"对应 `assignedAgentId`；§4 的 `spawnSession(opts)` 按 `agentId` 注入绑定，天然实现"子 Agent 继承所属数字员工的 skills/MCP 且不跨 Team"。

## 6. 技能（Skill）系统

### 6.1 现状（已核实）
- Pi Agent SDK 提供 `Skill` 类型（`name` / `description` / `filePath` / `baseDir` / `sourceInfo` / `disableModelInvocation`）、`loadSkills` / `LoadSkillsResult`、`parseSkillBlock` / `ParsedSkillBlock`。
- **运行时调用机制**：Agent 在消息中产出 `<skill name="...">` 块，`parseSkillBlock` 解析出技能名与位置，运行时加载对应 `SKILL.md` 指令并注入上下文由 Agent 遵循；`disableModelInvocation: true` 的技能**仅允许显式（斜杠命令）调用**，不被模型自动触发。
- 本仓库 `apps/web` 已有后端雏形：
  - `/api/skills/install`：`npx skills add <pkg> --agent pi`，支持 `global` / `project` 作用域；
  - `/api/skills/search`：搜索 skills.sh 市场（`https://skills.sh`）并回退到 `npx skills find`；
  - `/api/plugins`：用 `DefaultPackageManager` 按 `global` / `project` 解析 extensions + skills + prompts + themes，支持 install / remove / enable / disable。
- **缺口**：① 多租户——现有按 `cwd` + global/project 装载，无 team/user 维度；② 运行时接合——技能目录尚未按 tenant 喂给会话工厂；③ 前端——无 Vue3 技能界面。

### 6.2 技能格式与作用域
- 格式：复用 OpenClaw / Claude 的 `SKILL.md`（YAML frontmatter：`name` / `description` / 触发条件 + Markdown 指令体），与 MateClaw 技能格式互通。
- **隔离模型（已与用户决策，2026-07-16）**：默认 **「团队隔离 + 全局精选」，绑定单元下探到 Agent**——
  - `global`（全站精选库）：**仅管理员**可安装**通用、无凭证风险**的技能，全员可用；作为能力底座，不存放任何带密钥的技能。
  - `team`（团队预装层）：管理员/OWNER 把团队常用技能预装到 Team 作用域，团队内所有 Agent **默认继承**。
  - `agent`（数字员工，最终绑定单元）：每个 Agent 可在 team/global 基础上**显式授予白名单或排除**，实现"编码 Agent 用代码技能、写作 Agent 用 PPT 技能"的精细化。
  - `user`（个人，可选覆盖层）：个人安装的技能仅自己可见，非默认开启。**个人安装不强制属团队（userId 即可）；若含凭证/MCP 仍受 §7.3 铁律约束；纯指令型技能可完全 user 独立（B1 边界，2026-07-16）**。
  - 解析时某 Agent 的有效技能 = **四层继承**（global → team → user → agent，见 §5.4）后得出，**绝不跨 tenant 分发**（服务端按四层推导，禁止客户端指定目录）。

### 6.3 多租户装载（接缝）
- 在 `lib/rpc-manager.ts` 的 `spawnSession(opts)` 中，按 **`agentId`**（所属的数字员工）四层解析（global → team → user → agent，见 §5.4）出该 Agent 的有效技能集，再通过 `createAgentSessionServices` 的 `resourceLoaderOptions` 注入（`skillsOverride` 覆盖默认扫描结果 / `additionalSkillPaths` 追加团队·个人技能目录 / `noSkills` 显式禁用）。**注意：SDK 没有 `createAgentSessionServices({skills})` 这种平铺参数**，必须走 resourceLoaderOptions；这与 §4 的接缝一致。
- 复用现有 `DefaultPackageManager` 解析能力，但把"作用域"从 global/project 扩展到 global/team/user/agent，并**强制按 tenant 过滤**，避免跨租户技能泄漏。
- **SDK 兼容性说明**：Pi Agent SDK 原生的技能装载作用域只有 `global` / `project(cwd)` 两级，并不直接理解 team/user/agent。但我们**不需要 SDK 支持该粒度**——解析与过滤完全在应用层完成（查 `SkillPackage` / `AgentSkillBinding` / `UserSkillBinding`），只把最终算好的作用域（技能路径列表）通过 `resourceLoaderOptions` 注入会话。SDK 的 scope 机制只是可选捷径，我们走的是"自有解析 + 注入"路径（见 §4、§5.4）。SDK 零改动。

### 6.4 调用方式与界面（Vue3）
- **显式**：对话输入框支持 `/<skill>` 前缀（如 `/generate-ppt`、`/code-review`），映射到 `disableModelInvocation` 技能。
- **模型自决**：Agent 识别需求后自动产出 `<skill>` 块，前端给出可视化提示（如"正在使用 XX 技能"）。
- **技能中心页（新增 UI 屏）**：浏览 / 搜索市场、安装到团队 / 全局、启停、查看作用域与元信息；管理员可把团队常用 Skill 预装到团队作用域。

### 6.5 与 MCP / 多 Agent 的关系
- 技能是"指令 + 提示"层，MCP 是"工具"层；技能可编排调用多个 MCP 工具完成复合任务。
- 多 Agent 编排中，子 Agent 同样继承所属 tenant 的技能集，保证隔离一致。

## 7. MCP 工具生态设计

### 7.1 现状
`apps/web` **零 MCP**；`pi-mcp-extension` 为社区包，未安装（**C1 Spike**：验证 Node22 + Pi Agent 0.80.6 兼容性、stdio/SSE/Streamable HTTP 三传输、许可证；M4 前置，尽早开）。

### 7.2 引入方式
- 安装 `@earendil-works/pi-mcp-extension`（或社区等价包），通过 Pi Agent 的 `Extension` 系统注册 MCP Server。
- 支持传输：stdio / SSE / Streamable HTTP（以包实际支持为准）。

### 7.3 per-team / per-agent 绑定
- Pi Agent 核心**不自动隔离工具**，由调用方控制某会话拿到哪些工具。
- **绑定单元 = Agent（数字员工）**：在 `lib/tool-presets.ts` 按 **`agentId`** 四层解析（global 精选 → team 预装 → user 覆盖 → agent 白名单，见 §5.4）出该数字员工的有效 MCP Server 列表。MCP 扩展经 `createAgentSessionServices` 的 `resourceLoaderOptions.extensionsOverride`（或 `additionalExtensionPaths`）注入；工具级约束经 `createAgentSessionFromServices` 的 `tools` 参数。MCP 扩展随数字员工作用域加载，故须联动 §4 的「按作用域重建 services / 缓存键纳入作用域 hash」修正，否则 per-agent 绑定被击穿。
- 绑定关系存库（新增 `McpServer`、`AgentMcpBinding` 模型，见 §10；团队/用户预装层由 `McpServer.scope/teamId/userId` 表达，不再单独建 `McpBinding` 表以免双重表示），管理员/OWNER 在 Vue3 后台配置。
- **SDK 兼容性说明**：同 §6.3——Pi Agent 不原生理解 team/user/agent 隔离，工具注入由我们在应用层解析后通过 `resourceLoaderOptions` 传参完成，SDK 无需改动。
- **凭证隔离铁律（已决策，2026-07-16）**：MCP Server 常带凭证（Slack/GitHub token、DB 连接串等）。**带凭证的 MCP Server 严禁全局共享**；一律 `team` 或 `user` 作用域绑定，按 tenant 过滤只注入本租户会话。全局层仅允许挂载**无凭证**的公共服务（如只读 filesystem）。任何 MCP 凭证绝不跨团队可见。**（管理后台 UI 不得出现"绑定团队=全员"的带凭证 Server，如 slack——见 `admin-view.html` 已修正。）**

#### 7.3.1 参考 MateClaw 实测结论（刻意在此处分叉）
- 已读 `mateclaw-server` 真实源码（`tool/mcp/*`、`agent/binding/*`）。MateClaw 的做法是：**MCP Server 定义为全局、管理员级、凭证共享**——`MateMcpServer` 实体**无 workspace/owner 列**；`McpClientManager` 对每个 server 只保一个**全局共享的 `McpSyncClient`**，凭证（headersJson/envJson/command）写死在全局配置里；MCP 工具默认对**所有 agent 自动可见**（`getEffectiveToolNames` 中除非 `tools_disabled` 或 agent 显式勾选了特定 MCP 工具，否则全部注入）。
- 它的隔离是靠 `McpIdentityForwardService` 在**每次调用**时把"调用者身份"（on-behalf-of JWT / 明文 `<trust>:<subject>`）注入给上游 REST 后端做按人授权/审计——**但 MCP 连接本身的凭证是全局共用的管理员凭证**。
- 其代码自己承认：`MCP servers carry no workspace concept today`，任意 workspace 的 agent 都可绑定任意 MCP 虚拟技能。这是 MateClaw 作为"单组织数字员工平台"的有意取舍（所有 agent 默认可信）。
- **我们的项目是多团队、用户已明确担心跨团队凭证泄露，故刻意与 MateClaw 分叉**：MCP Server 定义加 `team_id`（或 `user_id`）作用域，连接/凭证**按租户隔离实例化**（Team A 的 Slack token 绝不进 Team B 的会话）；仅保留一个"全局精选、无凭证"的公共服务层（对标其 builtin 层）。可借鉴 MateClaw 的 **on-behalf-of 身份转发**思路：把调用用户身份随 MCP 调用带上，供上游后端做按人/按团队细粒度授权。

#### 7.3.2 参考 MateClaw 的 Skill 隔离（沿用其 workspace 模型）
- MateClaw 的 Skill 隔离分层清晰，值得沿用：**定义按 workspace 归属**（`SkillEntity.workspaceId`，默认 1 = builtin/全局），**绑定按 Agent（数字员工）**（`MateAgentSkill`：`agentId + skillId`，三段式：无行=继承全局默认 / 空集=显式无 / 非空=白名单），且**跨 workspace 绑定被显式拒绝（403）**（`requireSameWorkspace`）。
- 映射：我们的 `Team` ≈ 其 `Workspace`，我们的"用户工作空间/会话" ≈ 其 `Agent`。即：**技能定义按 Team 隔离、全局精选库全员共享；绑定可下探到 user/agent 级，并加跨 Team 拒绝守卫**。这正好印证了 PRD/设计已定的"团队隔离 + 全局精选"模型。

### 7.4 Tool Guard（审批/护栏）
- Pi Agent 0.80.6 暴露的钩子面为 `Extension` + `BashSpawnHook`；通用 `beforeToolCall` 三态审批（`pi-permission-system`）**需验证或自研（C3 Spike）**。本期至少实现"危险命令白名单 + bash 钩子 + MCP 工具按绑定可见"，完整审批流留 M5；若 `pi-permission-system` 可用则直接接入。

- `DelegationTree`：id、rootSessionId、parentSessionId、childSessionIds、mode(sync|parallel|async)、depth、status（**委派父子树专用表**，非 SessionManager.branch/fork）

- `PlatformApiKey`：id、provider、secret(加密存储)、updatedAt（平台级密钥池，供共享额度）
- `UserApiKey`（BYOK）：userId、provider、secret(加密)、updatedAt（用户个人密钥；按人隔离，优先于平台密钥）
- **凭证加密方式（D3，2026-07-16）**：MCP / 模型 API Key 一律**对称加密 `AES-256-GCM`**，主密钥（Secret encryption key）由环境变量持有；内部自用可接受，M5 迁移 KMS / Vault。绝不明文落库。

## 11. 风险与缓解

| 风险 | 缓解 |
|---|---|
| `pi-mcp-extension` 兼容性/许可证不明 | 先做 Spike 安装验证；审源码与协议（社区包） |
| Pi Agent 无原生多 Agent，Supervisor 自研复杂 | 复用 `DelegateAgentTool` + SessionManager 树；本期实现同步/并行/异步三模式，采用 MateClaw 实测护栏（深度≤3、并行≤8、结果截断 4000、子 Agent 禁用递归/记忆/改目标、事件回流、配额滚回）|
| 通用工具审批钩子缺失 | 先白名单 + bash 钩子；完整审批留 M5 |
| Vue3 与后端流式对接未验证 | 先做最小对接 spike，确认 `/api` 可被真实消费 |
| SQLite 多租户隔离弱 | 应用层强制 tenant 过滤兜底；Postgres+RLS 留 M5 |
| 技能/工具绑定需下探到 Agent（数字员工） | 新增 Agent 实体 + AgentSkillBinding/AgentMcpBinding/UserSkillBinding；spawnSession 按 agentId 四层解析（global→team→user→agent）后，经 `resourceLoaderOptions` 注入，SDK 不感知 team/user/agent |
| 配置被 AI 自动改导致死锁（已知坑） | 配置走人工/受控流程，最小权限 |
| 凭证明文存储 / 级联删除缺失 | MCP / 模型 API Key 一律加密存储（`secret` 字段）；删除技能 / MCP / Agent 时级联清理其绑定行；refresh token 置 httpOnly + CSRF 防护 |

## A. 跨文档术语与口径不一致（🔴 先对齐，避免实现反复）

### A1. 「并发数」与「活跃会话数」口径分裂
- **分歧**
  - PRD FR-06：本期"并发数即同时活跃会话数，不另设独立并发维度"。
  - `user-view.html`「我的设置 → 我的配额」同时列出 `会话数 5/5` 与 `并发数 3` 两个独立指标。
  - `admin-view.html` 团队配额用「并发数上限 10」；`index.html` 同一处却叫「会话数上限 5」。
- **影响**：配额维度数量、处罚口径、前端字段是否展示都依赖此定义。
- **建议默认值**：本期仅保留单一维度 **「同时活跃会话数」**，全文档统一术语；UI 中的「并发数」字段隐藏或改为「活跃会话数」的别名，不在 Prisma 新增独立并发列。

### A2. 「数字员工」被当成三种不同的东西
- **分歧**
  - `user-view.html` 数字员工 = 混合列表（团队预置可"克隆为个人" + 个人私有）。
  - `admin-view.html` 数字员工（模板）= 团队模板，可"设为团队默认"。
  - 设计 §5.4 的 `Agent` 实体 = 运行期绑定单元（`teamId / ownerUserId / scope`）。
- **影响**：决定要不要新增 `AgentTemplate` 表 / "克隆为个人"是复制一行实例还是引用模板。
- **建议默认值**：单一 `Agent` 表，用 `scope ∈ {team, personal}` 表达；"团队模板"即 `scope=team` 的 Agent 个人不可编辑（只读继承），"克隆为个人"= 复制一行新 `scope=personal` Agent 并允许改写绑定。不新增 `AgentTemplate` 实体（保持 YAGNI），但 UI 文案区分清楚"团队预置（只读）/ 个人（可编辑）"。

### A3. 登录入口是否统一
- **分歧**：设计文档强调"单一入口、所有角色同一界面"；`admin-view.html` 却有一张独立"平台管理后台"登录卡。
- **影响**：登录路由 / 权限收敛 / RBAC 实现方式。
- **建议默认值**：单一登录页，仅按 RBAC 渲染差异；删除独立管理员登录页，仅作设计稿角色区分示意保留但标注"非独立入口"。

## B. 直接决定 Prisma 数据模型的缺口（🟠 建议先定）

### B1. `user` 作用域技能的隔离边界
- **缺口**：FR-21 / §6.2 说 `user` 是"个人层、仅自己可见"；`user-view.html` 又说个人数字员工"技能/MCP 仅来自本团队已授权范围"。
- **问题**：个人安装的技能，是否必须先落在"本团队已授权"范围内？还是完全脱离团队？
- **影响**：`UserSkillBinding` 是否需要 team 校验守卫、四层继承解析顺序。
- **建议默认值**：个人安装技能**不强制**属于某团队（`userId` 即可），但若该技能含凭证/MCP，则仍受团队凭证隔离铁律约束；纯指令型技能可 user 独立安装。四层继承维持 `global → team → user → agent`。

### B2. 会话与工作区的绑定层级 + 多用户写同一 Project 的脏写
- **缺口**：FR-05 运行时 `cwd = user.lastProjectId → Project.rootPath`（文件按团队共享 Project）；但"会话按用户隔离"。
- **问题**：同团队两个用户用同一 Project 会各自起独立会话、写同一目录——文件冲突/脏写如何处理？Session 表是否同时挂 `userId`（隔离）+ `projectId`（cwd）？
- **影响**：Session schema、并发写保护策略。
- **建议默认值**：Session 同时挂 `userId + projectId`；本期**不对文件并发写做应用层互斥**（内部自用、低并发场景），仅在审计层记录写操作；后续 M5 视情况引入文件锁/分支隔离。文档需补一句"同目录并发写不做互斥"的明确取舍。

### B3. `AgentTemplate`（模板）层是否真的存在
- **缺口**：`admin-view.html` 有完整"数字员工模板 → 设为团队默认"流程；PRD/设计正文仅提"团队预装层"，未定义模板实体。
- **问题**：模板是独立可配置实体，还是"团队预装技能/MCP 集合"的别名？要不要持久化？
- **影响**：是否新增 `AgentTemplate` 表。
- **建议默认值**：同 A2，不新增 `AgentTemplate`；"模板"= `scope=team` 的 Agent，"设为团队默认"= 团队预装层（`SkillPackage.scope=team` / `McpServer.scope=team`）的快捷操作。

### B4. 邀请链接的目标用户与入职路径断点
- **缺口**：已决策"邀请链接不创建账号，被邀请者须先有平台账号"。
- **问题**：那么"外部人"实际入职路径是什么？必须管理员先开通 → 再发邀请？此时邀请链接相对"管理员直接拉人"的增量价值是什么？
- **影响**：`InviteLink.requireAccount` 语义、入职流程文案、是否保留邀请链接功能。
- **建议默认值**：邀请链接定位为"**把已有账号的跨团队/跨组织成员拉入本团队**"的自助入口（免去管理员逐个拉人），不面向全新外部人；全新外部人仍走"管理员开通账号 → OWNER 发邀请 → 用户接受加入"。文档需把"邀请链接 = 已有账号者的加入快捷方式"写明，消除"链接创建账号"的误解。

## C. 标注"待验证"、决定 M3/M4 能否启动的技术 Spike（🟢 与文档修订并行）

### C1. `pi-mcp-extension` 兼容性 Spike
- **出处**：PRD §9、设计 §7.1（已自标待 Spike）。
- **验证点**：Node22 + Pi Agent 0.80.6 是否可装可用、stdio/SSE/Streamable HTTP 三传输、许可证。
- **建议**：M4 前置，尽早开 Spike；若不兼容，需评估替代包或自研 Extension。

### C2. Vue3 ↔ `/api` 流式对接 Spike
- **出处**：设计 §9（待验证）。
- **验证点**：dashboard 当前 `src/mock` 占比、SSE/WS 事件格式是否与 Pi Agent 进程内事件对齐。
- **建议**：M3 最大风险点，最小对接 spike 优先做，确认 `/api` 可被真实消费后再铺功能。

### C3. 通用工具审批钩子能力面
- **出处**：设计 §7.4（"需验证或自研"）。
- **验证点**：`beforeToolCall` 三态审批（`pi-permission-system`）在 0.80.6 是否可用。
- **建议**：本期至少实现"危险命令白名单 + bash 钩子 + MCP 工具按绑定可见"，完整审批流留 M5；若 `pi-permission-system` 可用则直接接入。

## D. 安全/越权细节（🟢 内部自用可简化，但需写明取舍）

### D1. 审计日志颗粒度与保留期
- **缺口**：审计表记到什么级别（登录/改密/配额变更/调用 MCP/装技能）？保留多久？会否无限膨胀？
- **建议默认值**：本期审计记"身份/鉴权/配额/绑定变更 + MCP 调用 + 技能安装"级别（不记每次 token 级）；保留期默认 90 天，超期归档/清理。

### D2. 平台管理员能否读任意用户的会话内容
- **缺口**：权限矩阵只挡了"跨团队"，未明确"平台管理员 vs 任意用户会话内容"的可见性。
- **建议默认值**：平台管理员**仅可看会话元数据（标题/时间/token 用量）用于治理审计，不可读会话正文内容**；正文内容仅会话所属用户可见。文档需补这条隐私取舍说明。

### D3. BYOK / 平台密钥加密方式
- **缺口**：设计提到"加密存储"，未定算法与密钥管理（对称密钥放哪）。
- **建议默认值**：本期用对称加密（AES-256-GCM）+ 环境变量持有主密钥（` Secret encryption key`）；M5 迁移到 KMS / Vault。内部自用可接受。

## 处理建议

1. **第一批先收敛 A 组（术语对齐）+ B 组（数据模型缺口）** —— 它们直接决定 Prisma schema 与 UI 字段，定完即可起 M3 proposal。
2. **C 组 Spike 与文档修订并行**，不阻塞 A/B 的拍板。
3. **D 组记录取舍即可**，内部自用走简化路径，但要在设计文档补一句明确说明，避免实现时各人理解不一。
4. 本清单确认后，建议作为 M3 OpenSpec change 的 `proposal` 输入（`openspec-new-change` / `openspec-propose`）。
